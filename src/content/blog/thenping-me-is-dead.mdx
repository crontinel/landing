If you're a Laravel developer running scheduled tasks, you probably found **thenping.me** around 2020 or 2021. It was the first tool that actually understood the Laravel scheduler — hooking into Laravel's native events instead of just polling a URL like a generic cron monitor. It was elegant. It was Laravel-specific. And it worked.

**As of January 6, 2025, thenping.me is gone.**

The GitHub repository has been archived. The product is read-only. If you're still using it, you're running on borrowed time — no updates, no security patches, no support.

This is a practical look at what happened, why it matters for your Laravel apps, and what to do next.

## What thenping.me actually did

thenping.me solved a specific problem that generic cron monitors can't touch.

Laravel's scheduler (`php artisan schedule:run`) doesn't work like a traditional cron job. It runs every minute, reads your schedule definition, and decides which commands to fire based on their cron expressions. A command might run every five minutes, only on Tuesdays, or at 8am on the first of every month.

Generic monitors — the kind that just check whether a URL pinged back — can tell you *something* is wrong. But they can't tell you:

- Whether the scheduler itself ran
- Whether a specific scheduled command was supposed to fire at a given time
- Whether a command finished successfully or threw an exception
- Whether a job was skipped due to overlapping execution

thenping.me used Laravel's native event system to give you this granularity. It felt like a first-party Laravel package because, in many ways, it behaved like one.

## What went wrong

thenping.me was a solo or small-team operation. It never raised a round, never built a large team, and never expanded beyond Laravel scheduler monitoring.

The product simply stopped. No announcement. No farewell post. The GitHub repo went dark, and the last commits trailed off in September 2024 before the archive in January 2025.

If you search for it today, you'll still find articles recommending it. You'll still find the Laravel News mention from 2020. But the product is dead, and developers who built their monitoring around it are on their own.

## The problem with "good enough" monitoring

Many Laravel developers have cron monitoring that looks like this:

```bash
* * * * * curl -fsSL https://your-monitor.com/ping/abc123 >> /dev/null 2>&1
```

This tells you the scheduler *ran*. It does not tell you:

- The scheduler ran but all tasks were skipped because none were due
- The scheduler crashed before dispatching anything
- A specific high-value task failed and nobody noticed
- Queue workers stopped processing and never recovered

These are the failures that cost real money: the billing job that stopped processing invoices and nobody caught it until the next board meeting. The SSL renewal that silently failed for months. The webhook processor that dropped every event for two weeks.

Generic monitors give you a green checkmark when the real situation is red.

## Your options right now

### Healthchecks.io

The community default. Free tier, open source, widely recommended. Works for Laravel but doesn't understand the scheduler — it's a URL pinger, not a Laravel-native tool. You'll set up a ping for the scheduler heartbeat and that's it. No per-task visibility, no exit code capture, no schedule-aware missed-run detection.

### Spatie's laravel-schedule-monitor (open source)

If you want to stay self-hosted, Spatie's package logs every scheduled task run to your own database. Free, maintained, and built with proper Laravel event hooks. The catch: you're running your own monitoring stack. No alerting out of the box — you still need to check the logs, or wire up your own notification layer.

### Crontinel

We're the Laravel-native monitoring layer thenping.me was. Install in two commands, get per-task run history, failure alerts to Slack or email, and a dashboard that shows exactly what ran and when — not just whether your server sent a ping.

```bash

php artisan crontinel:install
```

We launched in 2026, built specifically for the way Laravel works: scheduler events, Horizon supervisor state, queue depth per queue. Not a generic monitoring tool with a Laravel skin.

Free tier covers one app with 7 days of history. No credit card required.

## The real lesson

The best monitoring for your Laravel app is the kind that knows what your Laravel app is supposed to be doing.

thenping.me understood that. That's why developers loved it. That's why its death leaves a real gap.

If you're still running thenping.me, it's time to move. If you're running a generic URL monitor, you're seeing only part of the picture. And if you've been putting off proper alerting because "nothing has failed yet" — the r/selfhosted and r/laravel threads full of silent failures discovered weeks too late are worth reading before that changes.

Your Laravel scheduler deserves more than a green checkmark that hides the truth.
